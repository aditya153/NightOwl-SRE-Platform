A Detailed Guide to NightOwl's MCP Architecture

A comprehensive, deeply humanized breakdown of exactly what we built on Days 12 and 13, why we built it, and exactly how it works under the hood.

🚨 The Ultimate Danger of Autonomous AI

To understand what we did on Days 12 and 13, we first have to understand the most terrifying problem in artificial intelligence today: Access.

The entire goal of the NightOwl platform is to have an AI system that acts like a human Site Reliability Engineer (SRE). If a server crashes at 3:00 AM, we want the AI to wake up, figure out what went wrong, and restart the server so the actual human engineers can keep sleeping.

The "Keys to the Kingdom" Problem

To allow an AI to restart a server, you must physically hand the AI the administrator passwords and security tokens to your company's entire infrastructure.

But AI algorithms are unpredictable. They are incredibly smart, but they are known to "hallucinate" or suffer from logic loops. If you give an AI raw administrator access, and it hallucinates, it might accidentally type Delete Production Database instead of Restart Web Server. It would instantly destroy the entire company with zero human oversight.

This risk is why most companies are terrified to let AI actually *do* things. We had to formulate a way to give the AI the hands it needs to fix things, while making it physically impossible for the AI to destroy things.

🛡️ The Solution: The Model Context Protocol (MCP)

To solve this massive security risk, we abandoned the idea of giving the AI direct passwords. Instead, we implemented Anthropic's brand new open-source standard: the Model Context Protocol (MCP).

Think of MCP as a Digital Bulletproof Glass Window between the AI and the company's servers.

  • On the left side of the glass sits the AI Brain. It has zero passwords. It cannot touch the servers. It cannot run code. It is completely isolated.
  • On the right side of the glass sits the actual servers, and a heavily guarded "Gatekeeper" (The MCP Tool Server).

When the AI figures out that a server needs to be restarted, it has to write down a formal, standardized request (a JSON document) and slide it through a tiny slot in the bulletproof glass to the Gatekeeper.

The Gatekeeper reads the note. The Gatekeeper has a strict, hardcoded list of the only things it is allowed to do. If the note says to do something safe (like restart a specific server), the Gatekeeper executes the command and passes the success receipt back through the glass. If the note asks to delete a database, the Gatekeeper instantly shreds the request and blocks it. The AI is perfectly sandboxed.

☸️ Day 12: Building the Kubernetes Gatekeeper

On Day 12, we built the Gatekeeper for our Kubernetes Infrastructure. Kubernetes is the engine that runs all the company's web services and backend applications.

1. Why did we build it?

Because Kubernetes is the most dangerous place for an AI to have direct access. If the AI had raw access to Kubernetes (via a standard KUBECONFIG file), it would have god-like powers over the infrastructure. By building a Kubernetes MCP Server, we created a localized firewall that strips the AI of that god-like power.

2. Exactly How did we build it?

We created an isolated Python script named kubernetes_server.py.

  • A We used a library called FastMCP to rapidly spin up the Gatekeeper server using the stdio (Standard Input/Output) transportation method.
  • B We explicitly programmed it to accept only two very specific tool requests: restart_pod and scale_deployment.
  • C Under the hood, when the Gatekeeper receives an approved request from the AI, it uses Python's native subprocess module to quietly run the exact safe terminal command, like kubectl delete pod <pod_name>.
# The exact code we wrote for the Restart tool: @mcp.tool() def restart_pod(namespace: str, pod_name: str) -> str: # Safely execute the exact terminal command subprocess.run(["kubectl", "delete", "pod", pod_name, "-n", namespace])

The Final Result: Even if the AI becomes entirely corrupted, the absolute worst thing it can do is restart a pod. It is physically incapable of formatting hard drives or wiping the cluster, because those actions are not programmed into the Gatekeeper. It is fundamentally secure by design.

🐙 Day 13: Building the GitHub Gatekeeper

On Day 13, we moved on to the second most sensitive part of a company: their Source Code and Github Repositories.

1. Why did we build it?

When the NightOwl AI discovers exactly which line of Python code caused a server to crash, we don't want it to just sit there in silence. We want the AI to automatically create a Pull Request (PR) with the fixed code, or write a detailed Comment on a GitHub Issue so the human engineering team can review it the next morning.

But again—if we gave the NightOwl AI the highly-classified GitHub Personal Access Token, it could accidentally delete the entire software repository or forcefully merge malicious code into production. We needed a second bulletproof glass window specifically for GitHub.

2. Exactly How did we build it?

We created a new isolated script named github_server.py.

  • A Securing the Secret Token: This script is the only thing that is allowed to look at the actual GITHUB_TOKEN environment variable. The AI never sees this token.
  • B Building the Gatekeeper Tools: We used FastMCP again to expose exactly two safe tools to the AI: create_pull_request and create_issue_comment.
  • C Making the Web Calls: Under the hood, the Gatekeeper uses Python's httpx library to securely format HTTP POST requests and beam them up to GitHub's formal REST API (https://api.github.com).
# The exact code we wrote for the Issue Comment tool: @mcp.tool() def create_issue_comment(owner: str, repo: str, issue_number: int, body: str) -> str: # 1. We load the secret token headers = get_github_headers() url = f"https://api.github.com/repos/{owner}/{repo}/issues/{issue_number}/comments" # 2. We securely post the AI's comment to GitHub httpx.post(url, headers=headers, json={"body": body})

The Final Result: The NightOwl Platform can now automatically document server crashes and propose code fixes directly on GitHub issues, all without having wildcard admin access to the repository.
